Manual: Implementacion de Endpoints POST y PUT en Loadout

Este documento explica paso a paso como se implementaron los endpoints POST /loadout y PUT /loadout en una arquitectura hexagonal con Spring Boot. Escrito para que cualquier persona pueda entender que se hizo, por que y como repetirlo en el futuro.

1. La arquitectura del proyecto

El proyecto usa arquitectura hexagonal (Ports & Adapters). La idea es separar el negocio (dominio) de los detalles tecnicos (HTTP, JSON, base de datos...).

CapaPaqueteQue contiene
Dominiodomain/Modelos de negocio y contratos (interfaces/puertos)
Aplicacionapplication/Logica de uso real: LoadoutService
Infraestructurainfrastructure/Controllers REST, DTOs, mappers, clientes externos
Regla de oro: el dominio NO conoce a la infraestructura. Nunca importes clases de infrastructure dentro de domain.

2. El flujo completo de una peticion

Cuando el cliente envia POST /loadout?idCharacter=1 con un body JSON:

Cliente HTTP
->
LoadoutController
->
LoadoutInputMapper
->
PostLoadoutUseCase
->
LoadoutService
->
CharacterClient / EquipmentClient
LoadoutService
-> Loadout ->
LoadoutResponseMapper
->
LoadoutResponseDto JSON
->
Cliente HTTP (201 Created)

Para PUT /loadout el flujo es identico pero pasa por UpdateLoadoutUseCase y devuelve 200 OK.

Paso 0 - El endpoint GET /loadout/{id} (base preexistente)

domain/port/input/GetLoadoutUseCase.java application/service/LoadoutService.java MODIFICADO infrastructure/input/rest/LoadoutController.java

El GET fue el primer endpoint implementado y sirvio de base para entender el flujo completo. A diferencia del POST y PUT, no necesita un modelo de entrada en el dominio: solo recibe un ID por la URL y devuelve un Loadout.

Puerto de entrada

La interfaz GetLoadoutUseCase define el contrato que el controlador usa para pedir un loadout. Solo necesita el ID del personaje:

public interface GetLoadoutUseCase {
    Loadout getLoadout(Long idCharacter);
}

Implementacion en LoadoutService

La logica es identica a la de POST/PUT pero sin recibir IDs de equipo: llama a los clientes externos, mapea los datos y construye el Loadout completo. En este proyecto, getLoadout reconstruye el loadout al vuelo (no hay base de datos); el ID del personaje se usa para buscar tanto al personaje como a su equipo en las APIs externas.

@Override
public Loadout getLoadout(Long idCharacter) {
    CharacterDTO characterDTO = characterClient.getCharacterById(idCharacter);
    List equipmentPieceDTOs =
        equipmentClient.getEquipmentPiecesByCharacterId(idCharacter);

    GameCharacter gameCharacter = characterMapper.fromDtoToDomain(characterDTO);
    Equipment equipment = equipmentMapper.toEquipment(equipmentPieceDTOs);

    BaseStats baseStats = new BaseStats(
        gameCharacter.getHealth(),
        gameCharacter.getAttack(),
        gameCharacter.getDefense(),
        gameCharacter.getSpeed()
    );

    FinalStats finalStats = getFinalStats(baseStats, equipment);

    return new Loadout(
        gameCharacter.getId(),
        gameCharacter.getName(),
        equipment,
        baseStats,
        finalStats
    );
}

Metodo en el controlador

El controlador extrae el ID de la URL con @PathVariable, delega en el caso de uso y convierte el resultado a JSON con el mapper de respuesta:

@GetMapping("/{id}")
public ResponseEntity getLoadout(@PathVariable Long id) {
    Loadout loadout = getLoadoutUseCase.getLoadout(id);
    return ResponseEntity.ok(loadoutResponseMapper.toResponse(loadout));
}

Por que GET no necesita LoadoutRequest ni LoadoutInput?
Porque no recibe datos de negocio del cliente: el unico parametro es el ID del personaje, que llega como parte de la URL (@PathVariable), no como un body JSON. Solo los endpoints que reciben datos complejos necesitan DTO de entrada y modelo de dominio de entrada.

Paso 1 - Crear el modelo de entrada en el dominio

NUEVO domain/model/LoadoutRequest.java

El dominio necesita su propio objeto para representar los datos que llegan en una peticion de creacion o actualizacion. No puede usar directamente el DTO del controlador porque eso crearia una dependencia del dominio hacia la infraestructura, rompiendo la arquitectura hexagonal.

Por eso creamos LoadoutRequest: un POJO del dominio con los IDs de las piezas de equipo elegidas.

package com.gamedev.domain.model;

import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@Data
@AllArgsConstructor
@NoArgsConstructor
public class LoadoutRequest {
    Long idArmor;
    Long idHelmet;
    Long idWeapon;
}
Por que no usamos el DTO del controlador directamente en el servicio?
Porque el dominio debe ser independiente del canal de entrada. Hoy los datos llegan por HTTP/JSON, pero manana podrian llegar por Kafka, gRPC o una CLI. El dominio no deberia cambiar por eso. LoadoutRequest pertenece al dominio; LoadoutInput pertenece a la infraestructura HTTP.

Paso 2 - Actualizar los puertos de entrada

MODIFICADO domain/port/input/PostLoadoutUseCase.java MODIFICADO domain/port/input/UpdateLoadoutUseCase.java

Un puerto de entrada es una interfaz Java que declara que operaciones ofrece la aplicacion. Es el contrato: el controlador solo sabe que existe este metodo, no como esta implementado.

Se actualizo la firma para recibir tambien el LoadoutRequest con los IDs del equipo:

// PostLoadoutUseCase.java
public interface PostLoadoutUseCase {
    Loadout postLoadout(Long idCharacter, LoadoutRequest request);
}

// UpdateLoadoutUseCase.java
public interface UpdateLoadoutUseCase {
    Loadout updateLoadout(Long idCharacter, LoadoutRequest request);
}
Antes de este cambio, la firma solo tenia Long idCharacter y no habia forma de saber que equipo queria equipar el personaje. Anadir LoadoutRequest completa la informacion necesaria.

Paso 3 - Implementar la logica en LoadoutService

MODIFICADO application/service/LoadoutService.java

LoadoutService implementa los tres puertos: GetLoadoutUseCase, PostLoadoutUseCase y UpdateLoadoutUseCase. Aqui vive la logica real.

Para postLoadout y updateLoadout el flujo interno es el mismo:

  1. Llamar a CharacterClient para obtener los datos del personaje por su ID.
  2. Llamar a EquipmentClient para obtener las piezas de equipo por sus IDs (los del request).
  3. Mapear los DTOs externos a modelos del dominio con CharacterMapper y EquipmentMapper.
  4. Construir BaseStats con las stats del personaje.
  5. Calcular FinalStats sumando los bonuses del equipo.
  6. Devolver un Loadout completo.
public Loadout postLoadout(Long idCharacter, LoadoutRequest request) {

    CharacterDTO characterDTO = characterClient.getCharacterById(idCharacter);
    List equipmentPieceDTO =
        equipmentClient.getEquipmentPiecesByIds(
            request.getIdArmor(),
            request.getIdHelmet(),
            request.getIdWeapon()
        );

    GameCharacter gameCharacter = characterMapper.fromDtoToDomain(characterDTO);
    Equipment equipment = equipmentMapper.toEquipment(equipmentPieceDTO);

    BaseStats baseStats = new BaseStats(
        gameCharacter.getHealth(),
        gameCharacter.getAttack(),
        gameCharacter.getDefense(),
        gameCharacter.getSpeed()
    );

    FinalStats finalStats = getFinalStats(baseStats, equipment);

    return new Loadout(
        gameCharacter.getId(),
        gameCharacter.getName(),
        equipment,
        baseStats,
        finalStats
    );
}

updateLoadout tiene exactamente el mismo cuerpo. La diferencia entre POST y PUT esta solo en el controlador (codigo HTTP de respuesta), no en la logica de negocio.

Paso 4 - Crear el DTO de respuesta (LoadoutResponse)

NUEVO infrastructure/input/rest/dto/LoadoutResponse.java

Un DTO (Data Transfer Object) es un objeto cuya unica funcion es transportar datos. En este caso, LoadoutResponse representa el JSON que devolvera la API al cliente.

Usa clases estaticas internas para anidar la estructura del JSON de forma limpia. Cada bloque del JSON (baseStats, equipment, finalStats) es una clase separada dentro de LoadoutResponse.

@Data
public class LoadoutResponse {
    Long characterId;
    String name;
    BaseStatsDTO baseStats;
    EquipmentDTO equipment;
    FinalStatsDTO finalStats;

    @Data
    public static class BaseStatsDTO {
        Integer health;
        Integer attack;
        Integer defense;
        Integer speed;
    }

    @Data
    public static class EquipmentDTO {
        WeaponDTO weapon;
        ArmorDTO armor;
        HelmetDTO helmet;

        @Data
        public static class WeaponDTO {
            String name;
            Integer weight;
            Integer attackBonus;
            Integer defenseBonus;
            Boolean isMagical;
        }
        @Data
        public static class ArmorDTO {
            String name;
            Integer weight;
            Integer defenseBonus;
        }
        @Data
        public static class HelmetDTO {
            String name;
            Integer weight;
            Integer defenseBonus;
            Integer focusBonus;
        }
    }

    @Data
    public static class FinalStatsDTO {
        Integer health;
        Integer attack;
        Integer defense;
        Integer speed;
        Integer focusBonus;
        Integer totalWeight;
    }
}

Paso 5 - Crear el mapper de respuesta (LoadoutResponseMapper)

NUEVO infrastructure/mapper/LoadoutResponseMapper.java

El mapper convierte el objeto de dominio Loadout en el DTO LoadoutResponse listo para serializar a JSON. Se usa MapStruct, que genera el codigo de mapeo automaticamente en tiempo de compilacion a partir de la interfaz.

El unico metodo que necesita implementacion manual es fromEquipmentDomainToDTO, porque MapStruct no puede deducir como transformar una List en tres campos separados (weapon, armor, helmet). Se implementa como metodo default:

@Mapper(componentModel = "spring")
public interface LoadoutResponseMapper {

    @Mapping(source = "characterName", target = "name")
    LoadoutResponse toResponse(Loadout loadout);

    LoadoutResponse.BaseStatsDTO fromBaseStatsDomainToDTO(BaseStats baseStats);
    LoadoutResponse.FinalStatsDTO fromFinalStatsDomainToDTO(FinalStats finalStats);
    LoadoutResponse.EquipmentDTO.WeaponDTO fromWeaponDomainToDTO(Weapon weapon);
    LoadoutResponse.EquipmentDTO.ArmorDTO fromArmorDomainToDTO(Armor armor);
    LoadoutResponse.EquipmentDTO.HelmetDTO fromHelmetDomainToDTO(Helmet helmet);

    default LoadoutResponse.EquipmentDTO fromEquipmentDomainToDTO(Equipment equipment) {
        if (equipment == null || equipment.getPieces() == null) return null;

        LoadoutResponse.EquipmentDTO dto = new LoadoutResponse.EquipmentDTO();
        for (EquipmentPiece piece : equipment.getPieces()) {
            if (piece instanceof Weapon w)       dto.setWeapon(fromWeaponDomainToDTO(w));
            else if (piece instanceof Armor a)   dto.setArmor(fromArmorDomainToDTO(a));
            else if (piece instanceof Helmet h)  dto.setHelmet(fromHelmetDomainToDTO(h));
        }
        return dto;
    }
}
Por que se usa @Mapping(source="characterName", target="name")?
El campo en el dominio se llama characterName pero en el DTO de respuesta se llama name. Con esta anotacion le decimos a MapStruct como hacer esa correspondencia.

Paso 6 - Crear el DTO de entrada HTTP (LoadoutInput)

NUEVO infrastructure/input/rest/dto/LoadoutInput.java

Este DTO representa el cuerpo JSON que el cliente envia en POST o PUT. Contiene exactamente los mismos tres campos que LoadoutRequest pero pertenece a la capa de infraestructura.

@Data
@AllArgsConstructor
@NoArgsConstructor
public class LoadoutInput {
    Long idArmor;
    Long idHelmet;
    Long idWeapon;
}

El JSON que envia el cliente tiene esta forma:

{
  "idArmor": 2,
  "idHelmet": 5,
  "idWeapon": 3
}

Paso 7 - Crear el mapper de entrada (LoadoutInputMapper)

NUEVO infrastructure/mapper/LoadoutInputMapper.java

Convierte el LoadoutInput (infraestructura) en LoadoutRequest (dominio). Esto es lo que permite que el dominio nunca vea el DTO HTTP. MapStruct genera todo automaticamente porque los nombres de campos son identicos.

@Mapper(componentModel = "spring")
public interface LoadoutInputMapper {
    LoadoutRequest toDomain(LoadoutInput input);
}

Paso 8 - Completar el controlador REST

MODIFICADO infrastructure/input/rest/LoadoutController.java

El controlador es el punto de entrada HTTP. Recibe peticiones, delega en los casos de uso y devuelve la respuesta. Usa @RequiredArgsConstructor de Lombok para inyectar las dependencias por constructor (buena practica frente a @Autowired).

@RestController
@RequestMapping("/loadout")
@RequiredArgsConstructor
public class LoadoutController {

    private final GetLoadoutUseCase getLoadoutUseCase;
    private final PostLoadoutUseCase postLoadoutUseCase;
    private final UpdateLoadoutUseCase updateLoadoutUseCase;
    private final LoadoutResponseMapper loadoutResponseMapper;
    private final LoadoutInputMapper loadoutInputMapper;

    @GetMapping("/{id}")
    public ResponseEntity getLoadout(@PathVariable Long id) {
        Loadout loadout = getLoadoutUseCase.getLoadout(id);
        return ResponseEntity.ok(loadoutResponseMapper.toResponse(loadout));
    }

    @PostMapping
    public ResponseEntity postLoadout(
            @RequestParam Long idCharacter,
            @RequestBody LoadoutInput input) {
        Loadout loadout = postLoadoutUseCase.postLoadout(
            idCharacter, loadoutInputMapper.toDomain(input));
        return ResponseEntity.status(HttpStatus.CREATED)
            .body(loadoutResponseMapper.toResponse(loadout));
    }

    @PutMapping
    public ResponseEntity putLoadout(
            @RequestParam Long idCharacter,
            @RequestBody LoadoutInput input) {
        Loadout loadout = updateLoadoutUseCase.updateLoadout(
            idCharacter, loadoutInputMapper.toDomain(input));
        return ResponseEntity.ok(loadoutResponseMapper.toResponse(loadout));
    }
}
AnotacionQue hace
@RestControllerMarca la clase como controlador REST (respuestas son JSON por defecto)
@RequestMapping("/loadout")Todos los endpoints de esta clase empiezan por /loadout
@GetMapping("/{id}")Mapea GET /loadout/{id}
@PostMappingMapea POST /loadout
@PutMappingMapea PUT /loadout
@PathVariableExtrae el {id} de la URL
@RequestParamExtrae ?idCharacter=1 de la query string
@RequestBodyDeserializa el JSON del body en el DTO
Diferencia clave entre POST y PUT:
POST devuelve 201 Created porque crea un nuevo recurso.
PUT devuelve 200 OK porque actualiza uno existente.
En este proyecto la logica de negocio es identica en ambos, la diferencia es solo el codigo HTTP.

11. Como repetir esto para un nuevo endpoint

Imaginemos que necesitas anadir DELETE /loadout. Estos son los pasos exactos:

  1. Si necesitas datos de entrada nuevos, crea un modelo en domain/model/ (ej. DeleteLoadoutRequest).
  2. Crea la interfaz del caso de uso en domain/port/input/ (ej. DeleteLoadoutUseCase).
  3. Implementa el metodo en LoadoutService e implementa la nueva interfaz.
  4. Si la respuesta tiene una forma nueva, crea un nuevo DTO en infrastructure/input/rest/dto/.
  5. Si hay mapeo nuevo, crea o actualiza un mapper en infrastructure/mapper/.
  6. Crea un DTO de entrada si el endpoint recibe un body (en infrastructure/input/rest/dto/).
  7. Crea un mapper de entrada si es necesario (en infrastructure/mapper/).
  8. Anade el metodo @DeleteMapping en LoadoutController.
Regla para recordar: cada vez que algo nuevo entra o sale por HTTP, necesita su propio DTO en infraestructura y su propio modelo en el dominio (si porta datos de negocio). Los mappers son el puente entre ambos mundos.
