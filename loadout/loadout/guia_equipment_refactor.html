Guía: Refactor de getFinalStats con interfaces y polimorfismo

El problema de partida

getFinalStats calculaba los stats finales accediendo a cada pieza de equipo por su nombre concreto:

// ❌ Versión original — acoplada a las clases concretas
private FinalStats getFinalStats(BaseStats stats, Equipment equipment) {
    Integer finalDefense =
            stats.getDefense()
                    + equipment.getHelmet().getDefenseBonus()
                    + equipment.getArmor().getDefenseBonus()
                    + equipment.getWeapon().getDefenseBonus();

    Integer finalAttack = stats.getAttack() + equipment.getWeapon().getAttackBonus();
    ...
}

Y Equipment era un contenedor con campos fijos:

// ❌ Versión original — no extensible
public class Equipment {
    Weapon weapon;
    Armor  armor;
    Helmet helmet;
}
¿Cuál es el problema? Si se añade una nueva pieza (Ring, Boots…) hay que volver a getFinalStats y añadir líneas manualmente. Esto viola el principio Open/Closed (OCP) de SOLID: el código debería estar abierto a extensión pero cerrado a modificación.

Paso 1 — Crear una interfaz común: EquipmentPiece

Concepto: Interfaz
Una interfaz es un contrato. Define qué métodos debe tener cualquier clase que la implemente, sin decir cómo los implementa. No tiene campos ni lógica propia. 
public interface EquipmentPiece {
    Integer getAttackBonus();
    Integer getDefenseBonus();
}
Toda pieza de equipo, sea la que sea, debe poder responder: "¿cuánto ataque das?" y "¿cuánta defensa das?".

Paso 2 — Las clases concretas implementan la interfaz

Concepto: implements vs abstract
  • implements → La clase se compromete a tener los métodos de la interfaz. Sigue siendo concreta e instanciable con new Armor(...).
  • abstract → La clase NO se puede instanciar. Existe solo para ser extendida. No es lo que necesitamos aquí.

Weapon

Ya tenía attackBonus y defenseBonus como campos. Lombok genera los getters automáticamente, así que solo añadiendo implements EquipmentPiece cumple el contrato sin cambiar nada más.

public class Weapon implements EquipmentPiece {
    private Integer attackBonus;   // → Lombok genera getAttackBonus()
    private Integer defenseBonus;  // → Lombok genera getDefenseBonus()
    ...
}

Armor y Helmet

No tienen attackBonus (no atacan). Hay que implementar el método manualmente devolviendo 0. El campo no hace falta, solo el método.

public class Armor implements EquipmentPiece {
    private Integer defenseBonus;  // → Lombok genera getDefenseBonus()

    @Override
    public Integer getAttackBonus() {
        return 0;  // La armadura no da bonus de ataque
    }
}
Concepto: Polimorfismo
Las tres clases son distintas pero todas responden al mismo mensaje (getAttackBonus(), getDefenseBonus()). Cada una lo hace a su manera. Esto es polimorfismo: mismo método, comportamiento diferente según el tipo real del objeto.

Paso 3 — Equipment usa List

En lugar de campos fijos, Equipment contiene una lista de cualquier cosa que sea un EquipmentPiece.

public class Equipment {
    List<EquipmentPiece> pieces;
    // Puede contener Weapon, Armor, Helmet, o cualquier pieza futura
}
Para construirlo: new Equipment(List.of(weapon, armor, helmet))
