GAMEDEV — Documentation (EN)

Author: Andrés Pestana Hidalgo

This guide explains the project in a clear and direct way: first technologies and composition, then how each microservice works.

1) What is GAMEDEV

GAMEDEV is a project split into three microservices:

  • characters: manages characters (name and base stats).
  • equipment: manages equipment pieces (weapon, armor and helmet).
  • loadout: the integrator that consumes characters + equipment to build a “loadout” (character + full equipment + final stats).

Main idea

loadout does not “invent” data: it fetches it via HTTP from the other microservices and then calculates the final result.

2) Technologies used

  • Java 17
  • Spring Boot (web / MVC, dependency injection)
  • Maven (build & run)
  • MapStruct (DTO ↔ domain mapping)
  • Lombok (reduces boilerplate)
  • PostgreSQL (real persistence in characters and equipment)
  • DBeaver (client to inspect/manage PostgreSQL)
  • OpenAPI/Swagger (specification in loadout)
  • Tests with Spring Boot Test

What matters is that Spring Boot exposes HTTP endpoints, and loadout calls other endpoints.

3) Repository structure

There are three main folders (one per microservice):

  • characters/characters
  • equipment/equipment
  • loadout/loadout

Each folder is an independent Maven project with its own pom.xml.

4) Ports and base URLs

Microservice Port Base URL Role
characters 8080 http://localhost:8080 Character data
equipment 8081 http://localhost:8081 Equipment data
loadout 8082 http://localhost:8082 Integrator / orchestration

In loadout, the consumed services are configured as: http://localhost:8080 and http://localhost:8081.

5) How to run locally

The simplest approach is to open 3 terminals and run:

Requirement: PostgreSQL + DBeaver

  • PostgreSQL must be running at localhost:5432.
  • Databases: characters_db and equipment_db.
  • Use DBeaver to connect to the server and press Refresh to see tables/data.

If tables do not show up in DBeaver, ensure you do not have an active filter (it appears in the tree as “Filtered by search”) and press F5 to refresh.

characters

PowerShell

cd C:\dev\gamedev\characters\characters
mvn spring-boot:run

CMD

cd /d C:\dev\gamedev\characters\characters && mvn spring-boot:run

equipment

PowerShell

cd C:\dev\gamedev\equipment\equipment
mvn spring-boot:run

CMD

cd /d C:\dev\gamedev\equipment\equipment && mvn spring-boot:run

loadout

PowerShell

cd C:\dev\gamedev\loadout\loadout
mvn spring-boot:run

Important: do not use && in PowerShell. It is allowed in CMD.

If a microservice does not start, ensure the port is free and Java 17 is installed.

6) Internal architecture (Hexagonal) and SOLID principles

This project follows a hexagonal architecture (Ports & Adapters) approach. The main goal is to decouple business logic from technical details (web framework, database, or HTTP calls).

Layers and responsibilities

  • domain: business models and rules (plus domain exceptions). It should not depend on frameworks.
  • domain/port: ports (interfaces) that define what the domain needs and provides.
  • application: use cases/services that orchestrate domain logic.
  • infrastructure: input/output adapters (REST controllers, persistence, HTTP clients).

SOLID (in practice)

  • S (Single Responsibility): each class focuses on one job (controller, service, mapper, client...).
  • O (Open/Closed): behavior is extended via new adapters/implementations without breaking the core.
  • L (Liskov Substitution): implementations respect the contracts defined by ports (interfaces).
  • I (Interface Segregation): small, focused ports instead of “god” interfaces.
  • D (Dependency Inversion): the domain depends on abstractions (ports), not on details.

7) Loadout: the integrator

loadout is the most important microservice in terms of “flow”, because it:

  • receives the client request (for example: “get character 1 loadout”)
  • calls characters to fetch the character
  • calls equipment to fetch equipment details
  • computes final stats (e.g., attack/defense/total weight)
  • returns a unified response

Consumed services configuration

File: loadout/loadout/src/main/resources/application.yaml

8) Loadout API (quick summary)

The specification is located at: loadout/loadout/src/main/resources/openapi.yaml.

  • GET /characters/{id}/equipment → returns character + equipment + final stats
  • POST /characters/{id}/equipment → assigns chosen equipment to a character
  • PUT /characters/{id}/equipment → edits an existing loadout

Example (conceptual)

GET http://localhost:8082/characters/1/equipment

The exact JSON is defined in OpenAPI. It includes: baseStats, equipment and finalStats.

9) Common errors & tips

  • 404 Character not found: the character does not exist in characters.
  • 400 Invalid input: the request body does not match the expected model.
  • Always start in this order: characters → equipment → loadout.
  • If you change ports, update loadout's application.yaml.
